# Stale plan detail not removed from bodyRow when switching plans in TUI

**Date:** 2026-06-28 18:22
**Files:** packages/lazyaif/src/views/plans-viewer/tui-view.ts
**Severity:** medium

## Problem

In the TUI plans viewer, switching between plans in the left `Select` list did not
reliably show the selected plan in the right detail pane. The previously displayed plan's
detail `ScrollBox` could remain visible (overlapping the new one), so the user saw the wrong
plan after switching.

## Root Cause

Two compounding issues in `createPlansTuiApp`:

1. **Initial detail never tracked.** The initial detail renderable was added to `bodyRow`
   via a standalone `bodyRow.add(renderTaskDetail(...))` call, but the return value was
   discarded — `currentDetail` stayed `null`. As a result, the first `SELECTION_CHANGED`
   event hit the `if (currentDetail)` guard (false), skipped disposal entirely, and simply
   added a *second* detail `ScrollBox` on top of the initial one. Both occupied the same
   `width: "60%"` slot and overlapped.

2. **`dispose()` does not detach from the parent.** Even for subsequent switches where
   `currentDetail` was tracked, `updateDetail` only called `dispose()` on the previous
   detail. Per opentui's renderable-tree model, `dispose()` cleans up internal resources
   but does **not** remove the node from its parent's child list. The old `ScrollBox`
   therefore lingered in `bodyRow`. opentui's documented API for removing children is
   `container.remove(child)` (see `docs/core-concepts/renderables.mdx` and the
   `tab-select.mdx` example).

The net effect: every switch stacked a new detail without removing the old one, so the
previous plan stayed visible / bled through — "switching plans shows the wrong one".

## Solution

- Route the initial render through `updateDetail()` so `currentDetail` is always tracked
  from the start.
- In `updateDetail`, explicitly `bodyRow.remove(currentDetail)` *before* disposing it and
  before adding the new detail.
- Add a bounds check on the `SELECTION_CHANGED` index to guard against any opentui index
  semantics that could emit values outside `[0, plans.length)` (e.g. end-of-list navigation
  with `wrapSelection: false`). Out-of-range indices now log a warning and keep the current
  detail rather than leaving the pane empty.
- Add `[tui:app]` debug/warn logs around removal, mount, and bounds checks so the behavior
  can be verified at runtime.

## Prevention

- When swapping one child renderable for another in opentui, always `parent.remove(old)`
  in addition to (or instead of) `old.dispose()`. `dispose()` alone does not detach from
  the parent.
- Always record the currently mounted child reference (including the initial render) so
  cleanup can find it. A common pattern: a single `setDetail()` / `updateDetail()` helper
  used for both initial mount and subsequent swaps.
- When wiring a component event to an array-indexed state, validate the index bounds
  before using it — UI component event payloads are not guaranteed to match data indices.
- For TUI bugs where "switching shows the wrong/old item", suspect stale-children-not-removed
  first: check that the old child is actually detached from the parent, not just disposed.
