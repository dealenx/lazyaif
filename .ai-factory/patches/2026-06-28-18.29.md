# Enter не переключал план; старый detail не удалялся из bodyRow

**Date:** 2026-06-28 18:29
**Files:** packages/lazyaif/src/views/plans-viewer/tui-view.ts
**Severity:** high

## Problem

В TUI plans-viewer Enter при выборе плана не работал — правая детальная панель всегда
показывала первый план. Стрелки также не обновляли содержимое детали.

## Root Cause

Две независимые ошибки в `renderPlanList` и `updateDetail`:

1. **Enter не обрабатывался.** Подписка была только на
   `SelectRenderableEvents.SELECTION_CHANGED` (срабатывает на стрелки Up/Down —
   перемещение подсветки). Событие `SelectRenderableEvents.ITEM_SELECTED`
   (`"itemSelected"`, срабатывает по Enter/linefeed) **не было подключено**.
   Согласно исходнику `@opentui/core` (`index.js:10868`), именно `selectCurrent()`
   эмитит `itemSelected`, а `moveUp`/`moveDown` эмитят `selectionChanged`. Без
   подписки на `ITEM_SELECTED` Enter просто некому было обработать.

2. **Старый detail не удалялся из `bodyRow`.** API `Renderable.remove(id: string)`
   принимает **строковый id**, а не объект-рендерабл (см. `Renderable.d.ts:270`:
   `remove(id: string): void` и `index.js:3438`: `super.remove(this.target.id)`).
   Предыдущий фикс передавал `bodyRow.remove(currentDetail)` — это молча не
   удаляло ничего (ошибка попадала в `try/catch { /* noop */ }`). В результате
   первый (индекс 0) detail, смонтированный при старте, оставался в дереве
   навсегда, и все новые детали рисовались поверх него, перекрываясь layoutsом.

   Корень ошибки — неверное прочтение документации: в
   `docs/core-concepts/renderables.mdx` пример `container.remove("body")` явно
   показывает строковый id, но мы передали объект.

## Solution

- Подписались на `SelectRenderableEvents.ITEM_SELECTED` дополнительно к
  `SELECTION_CHANGED`, чтобы Enter (а также linefeed) обновлял детальную панель.
- В `updateDetail` исправили вызов удаления:
  `bodyRow.remove(currentDetail.id)` вместо `bodyRow.remove(currentDetail)` —
  передача строкового id согласно сигнатуре `Renderable.remove(id: string)`.
- Усилили диагностику: логируем id удаляемого/монтируемого detail, а в случае
  ошибки `bodyRow.remove` выводим `console.warn` (раньше ошибки глотались молча).

## Prevention

- При работе с API opentui сверяться с сигнатурами в `.d.ts`, а не только с
  примерами в mdx. `remove(id: string)` строго требует строку — передача объекта
  молча не отсоединяет дочерний узел.
- TUI-компоненты с раздельными событиями "highlight changed" и "item selected"
  требуют подписки на оба, если нужно обновлять и превью, и подтверждённый
  выбор. Не путать `SELECTION_CHANGED` (highlight) и `ITEM_SELECTED` (enter).
- `try/catch` с пустым телом глотает реальные ошибки API и маскирует баги.
  Хотя бы `console.warn` в catch делает проблему видимой в логах.
